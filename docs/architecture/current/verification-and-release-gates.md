# Verification and Release Gates

> Status: **Runtime Authority**

## 1. Objective

Verification determines whether evidence satisfies a declared contract for an exact repository state. Release gating determines whether the repository is ready for an externally visible delivery action.

The system distinguishes:

```text
execution success
integration success
verification success
Task acceptance
Issue completion
release readiness
release execution
```

No step implies the next without its required evidence.

## 2. Evidence Principles

Evidence must be:

- generated by the Controller or a trusted repository system;
- bound to stable repository and checkout identity;
- bound to an exact Revision or artifact hash;
- addressable through a durable artifact reference;
- bounded when returned to clients;
- honest about failure, timeout, truncation, or missing data;
- reusable only while its cache/precondition contract remains valid.

Worker or Agent prose is supplementary evidence, not authoritative pass/fail proof.

## 3. Verification Record

A Verification Record contains:

```text
repoId
checkoutId
revision
issueId / taskId
runId / editSessionId / integrationId
reviewedDiffHash
checkResults
commandEvidence
acceptanceResults
reviewer
verifiedAt
environmentFingerprint
artifact references
staleAt / staleReason when invalidated
```

A caller cannot satisfy required named checks by reporting `ok: true`. The Controller executes the check or verifies a trusted external artifact.

## 4. Exact Revision

Verification applies to one Revision definition, normally derived from:

```text
HEAD
tracked working-tree diff
relevant untracked files
check configuration
environment fingerprint
```

The Revision contract excludes known runtime-only evidence directories.

If relevant files change after verification:

- the previous record remains historical;
- completion logic treats it as stale;
- required verification runs again for the new Revision.

## 5. Verification Layers

### Layer 1 — Patch Integrity

For Direct Edit or integration:

- allowed paths respected;
- before/after hashes recorded;
- diff is available;
- change budget respected;
- no unsupported binary/rename behavior hidden;
- user changes outside scope preserved.

### Layer 2 — Focused Checks

Run the smallest deterministic checks that directly cover the change.

Examples:

- one test file;
- one package type check;
- one domain test suite;
- one architecture or documentation gate.

### Layer 3 — Task Contract

Evaluate all Task-declared checks and acceptance criteria.

Acceptance criteria must be observable and must not merely repeat “implementation complete.”

### Layer 4 — Integration Verification

After an isolated Run is integrated:

- verify target Revision, not only Worktree Revision;
- verify integration record and reviewed diff hash;
- rerun affected checks when target state changed;
- retain Worktree until durable integration outcome exists.

### Layer 5 — Issue or System Integration

Run broader checks required by the Issue, architecture boundary, or affected runtime subsystem.

### Layer 6 — Release Gate

Run release-specific checks against a frozen candidate Revision.

## 6. Risk-Adaptive Verification

### Read-only

Requires evidence of the requested observation. No diff gate.

### Low-risk change

Focused checks when declared; may auto-complete after successful applicable evidence.

### Medium-risk change

Declared checks and acceptance evidence apply. Auto-completion is allowed only when policy says no human decision is required.

### High-risk change

Requires:

- reviewed diff or integrated Revision;
- required verification evidence;
- acceptance criteria results;
- explicit human acceptance.

### Destructive change

Requires all high-risk evidence plus explicit same-request destructive authorization and operation-specific safety/rollback plan.

## 7. Task Acceptance

Task acceptance occurs after verification policy passes.

Acceptance action records:

```text
acceptedBy
acceptedAt
verificationId
revision
note
```

Dependent Tasks unlock only from accepted terminal semantics, not from a successful Run alone.

For eligible low/medium-risk work, Controller policy may auto-accept and records that automation explicitly.

## 8. Failed Verification

A real failed check or acceptance criterion moves work to `changes_requested` or a similarly explicit corrective state.

Failure handling:

1. preserve check output and artifact;
2. classify failure as introduced, pre-existing, environmental, flaky, stale, or contract-related;
3. avoid claiming unrelated pre-existing failure was caused or fixed;
4. create a focused correction or blocker;
5. rerun only the necessary evidence after the Revision changes.

A flaky check requires explicit flake evidence and retry policy. Repeating until green without recording attempts is forbidden.

## 9. Shared Check Evidence

Identical check requests may share one execution when these match:

```text
repoId
checkoutId
checkId
revision
timeout contract
environment fingerprint
```

Every subscriber links to the same immutable evidence. One subscriber cancellation must not terminate execution required by others.

## 10. Release Readiness

A repository is `release_ready` only when all required conditions are true for the candidate Revision.

Baseline gate:

```text
repository identity valid
remote/GitHub mapping diagnosed
candidate Revision fixed
Workspace ownership known
no active write Job
no active Agent Run
no pending integration
no unresolved integration conflict
no dirty Edit Session
no orphan/unknown execution affecting candidate
required Tasks accepted
required checks pass for candidate Revision
architecture/document gates pass
package/public metadata valid
release manifest generated
rollback or recovery plan recorded when required
human authorization boundary identified
```

A project may add stricter repository-specific gates.

## 11. Release Freeze

Before final release verification, the Repo Actor acquires an exclusive release Claim and enters Release Freeze.

During freeze:

- new mutation Jobs wait;
- active writers finish or are explicitly resolved;
- Integration Queue drains;
- read-only inspection continues;
- release checks run against the fixed candidate;
- Schedule mutation work pauses;
- any relevant file change invalidates the candidate and requires a new gate.

Release Freeze is released after:

- release execution completes;
- release is cancelled;
- verification fails and the candidate returns to development;
- Lease expires and recovery reconciles state.

## 12. Release Manifest

The release gate produces a durable manifest:

```text
repoId
checkoutId
candidateRevision
branch
target version or release identifier
included Issues and Tasks
verification records
check artifact hashes
known warnings
known non-blocking pre-existing failures
remote target
release actions requiring authorization
rollback/recovery instructions
createdAt
approvedBy / approvedAt when applicable
```

The manifest is an execution input and audit record. It does not execute release actions by itself.

## 13. External Actions

The following remain explicit external boundaries:

- commit when user policy requires review;
- push;
- create/update remote PR when not already authorized by the Task;
- merge;
- tag;
- package publication;
- production deployment;
- remote Issue closure;
- destructive migration or data mutation.

The Controller may prepare exact commands and evidence. It may execute only with explicit authorization appropriate to the operation.

## 14. Release Job

Release execution, when authorized, is a durable Job.

It records:

```text
releaseManifestId
requestId
remote target
operation sequence
current step
external identifiers
stdout/stderr/artifacts
partial outcome
compensation or resume instructions
```

A disconnected client does not make external outcome unknown if provider or command evidence can prove it. Otherwise the Release Job becomes orphaned/unknown and requires reconciliation before retry.

## 15. Partial Release

Multi-step or multi-repository release may end partially complete.

Partial outcome records:

- completed external actions;
- failed or unstarted actions;
- current deployed/published versions;
- safe compensation options;
- actions that require human decision.

The system must not collapse partial release into success or rerun all actions blindly.

## 16. Rollback and Recovery

Rollback is not assumed universally safe.

A release plan declares one of:

```text
reversible with deterministic rollback
forward-fix preferred
feature-flag disable
provider-native rollback
manual recovery only
```

Rollback itself is externally visible/destructive work and requires appropriate authorization and evidence.

## 17. Architecture Gate

Architecture-sensitive releases must pass:

- current architecture document presence;
- Runtime Authority marker;
- historical document marker policy;
- architecture drift check;
- relevant ADR linkage;
- migration roadmap update when target gaps change.

Architecture documentation is part of the release contract, not an optional afterthought.

## 18. Verification Performance

Verification must remain proportional:

- focused checks during iteration;
- Task checks before acceptance;
- broader Issue/system checks at integration boundaries;
- full release gate once the candidate is frozen.

Running the full release suite after every small patch wastes resources and increases queue contention. Running only focused checks before release is insufficient.

Check results and revision-aware caches prevent unnecessary duplication without hiding stale evidence.

## 19. Current Implementation

Verification evidence is bound to Job, repository and exact Git revision with an environment fingerprint. Revision changes invalidate reusable evidence. Oversized result bodies are stored as Artifacts and only bounded previews are returned through Job status.

Release readiness is implemented as a durable `release-gate` Job holding an exclusive `release:<repoId>` Lease. The Gate checks Workspace cleanliness, active Execution Jobs, Agent Runs, Local compatibility Jobs, pending Worktree integration, unfinished Edit Sessions, other Leases, active-Issue Tasks, exact-revision verification, repository remote identity, GitHub mapping, Controller Daemon readiness and package metadata.

A successful Gate returns a release manifest for the exact current revision. Push, merge, publish, deploy and destructive external operations still require a separate explicit user authorization.

## 20. Required Release Tests

Runtime and release checks cover:

- candidate revision changes invalidate evidence;
- active writers block release;
- pending Worktree integration blocks release;
- dirty Edit Sessions block release;
- remote and GitHub mapping drift block release;
- release Lease blocks new writes;
- a successful Gate does not execute an external side effect;
- partial external outcomes never become global success.

## 21. Immutable Check Definition Snapshot

A scheduled check stores the normalized command, cwd, timeout, source, registry revision, and definition digest at admission time. Execution validates the digest and repository-relative cwd, then uses that snapshot even if another checkout or a later commit changes the check registry.

The snapshot stabilizes check identity; the produced evidence is still bound to the revision observed when execution starts and becomes stale if repository content changes while the command runs. This prevents both `check not found` after checkout switching and accidental reuse of evidence from a different repository revision.
