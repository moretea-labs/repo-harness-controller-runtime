# 教程 3：完成第一个仓库任务

第一次运行请选择一个小型文档或配置修改。目标是先验证仓库路由、有边界编辑、审查证据和清理流程，再处理复杂代码。

## 1. 先只读检查

对 ChatGPT 说：

```text
使用 repo-harness。先检查 rh_status，再为我选择的仓库加载 rh_context。总结当前分支、工作区状态、相关文件和可用命名检查，暂时不要修改。
```

如果工作区不干净或仓库目标不明确，先处理这些问题再写入。

## 2. 给出有边界任务

示例：

```text
更新 docs/example.md 中的安装说明。修改只限这个文件，展示 diff，并运行最相关的文档命名检查。不要 push，也不要发布。
```

好的首个任务应写清目标、允许范围、验证方式和远程写入边界。

## 3. 理解执行方式

小而明确的修改应优先 Direct Edit。它会记录文件 fingerprint、revision、持久 diff 和检查证据。范围较大、存在依赖或需要跨会话恢复时，才升级为持久 Issue/Task/Run。编码 Agent 是可选执行者，不是默认总控。

## 4. 接受前审查

确认：

- 只修改了预期文件；
- diff 与需求一致；
- 命名检查真实执行且通过；
- 没有加入凭据、日志、运行态或个人绝对路径；
- 没有未经授权的远程动作。

需要修改时通过 `rh_work` 打回；需要明确决定或审批时查看 `rh_inbox`。

## 5. 提交与远程动作

本地 commit 可以属于已授权仓库流程，但 push、创建 PR、修改 GitHub、打 tag 和发布都是独立外部动作。应明确写出边界，例如“只在本地提交，不要 push”。

## 6. 确认清理和可恢复性

结束时要求返回：

- 最终 Git 状态；
- 改动文件和检查；
- 如果要求提交，则返回 commit hash；
- 是否还存在 worktree、branch、Job 或 attention item；
- 下一步安全动作。

完成的本地任务应留下干净工作区，或明确标识其他并行任务的改动。日志和执行证据只保存在被忽略的 Controller 运行态中。

开启 Agent、GitHub、Browser、Schedule 或 Workspace 插件前，先阅读[功能与配置层级](../operations/features.zh-CN.md)。
