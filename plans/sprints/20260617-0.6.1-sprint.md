有，而且现在的优化空间已经从“补基础架构”变成了 **把 0.6 的新架构产品化、收口默认路径、减少长期维护面**。我不会建议推倒重来；当前 0.6 的方向是对的：`src/core/adoption`、`src/effects`、process runner、planner/applicator 已经把边界拉开了。下一阶段应围绕 **事务闭环、默认 apply 迁移、manifest 单一事实源、adapter 插件化、发布工程闭环** 来做。

先提醒一个小问题：当前仓库里的 0.6.0 release checklist 仍写着 “npm publish is blocked by missing npm auth”，并且保留 “Do not create v0.6.0 tag or GitHub release until npm publish…” 的 hold 记录。若你已经发布了 0.6，这个文件需要更新为最终发布证据。

## 1. 优先级最高：Transaction Manifest + Rollback Executor

现在的 0.6 已经有 rollback metadata、atomic writer、backup path，但 rollback 仍然是“可审计说明”，不是可执行恢复。架构文档也明确写了 rollback metadata 不执行 rollback。

下一步最有价值的是引入真正的 transaction manifest：

```text
.ai/harness/backups/fs-transaction/<timestamp>/manifest.json
```

每次 `adopt --experimental-ts-apply` 应该写入：

```json
{
  "protocol": 1,
  "command": "adopt",
  "created_at": "2026-06-16T...",
  "repo_root": "...",
  "mode": "standard",
  "operations": [
    {
      "id": "appendManagedBlock:.gitignore:repo-harness-generated-runtime",
      "kind": "appendManagedBlock",
      "path": ".gitignore",
      "status": "applied",
      "backupPath": ".ai/harness/backups/fs-transaction/..."
    }
  ],
  "rollback": {
    "command": "repo-harness adopt rollback --transaction .ai/harness/backups/fs-transaction/<timestamp>/manifest.json"
  }
}
```

然后新增：

```bash
repo-harness adopt rollback --transaction <path>
```

Rollback 规则：

```text
writeFile ifMissing -> 若文件 hash 仍匹配创建内容，则删除
writeFile replacement -> 恢复 backup
appendManagedBlock -> 恢复 backup
mkdir -> 仅当目录为空时删除
unsupported/manual -> 拒绝自动 rollback，要求人工 review
```

为什么这是下一阶段最优先：0.6 已经有 `atomicWriteFile()`、target lock、backup、fsync 和 preflight；只差把这些 apply results 组织成可执行恢复链。`fs-transaction.ts` 当前已经负责 backup、lock、temp write、rename 和 fsync。

## 2. 把 `--experimental-ts-apply` 分阶段转正

当前文档清楚说明 default `adopt --mode standard` 仍走 shell migrator，`minimal/self-host` 在非 TS dry-run 或 `--experimental-ts-apply` 下 fail-closed。 这是正确的保守策略。

但 0.6 之后应该规划转正路径：

### 0.6.x

```text
--experimental-ts-apply 继续 opt-in
补 transaction manifest + rollback executor
补 install-from-tarball smoke
```

### 0.7.0

```text
新增 --ts-apply 作为非 experimental alias
保持 --experimental-ts-apply 兼容
对 standard downstream repo 做 shell-vs-TS parity report
```

### 0.8.0

```text
standard downstream repo 默认 apply 走 TS applicator
shell migrator 退为 --legacy-shell-apply
self-host / repo-pinned runtime 仍需显式模式
```

这样不会突然破坏用户仓库，也能逐步让 0.6 的架构成果变成默认路径。

## 3. Runtime Manifest / Workflow Contract 继续下沉为单一事实源

当前已经把 bootstrap templates 移到 workflow contract manifest，planner 只渲染 `{{repoName}}`。文档里也记录了这一点。

下一步可以继续把这些内容从代码常量迁出：

```text
required directories
required files
helper wrapper policy
gitignore managed block entries
operation risk level
rollback strategy default
mode surface: minimal / standard / self-host
```

现在 `plan.ts` 里仍有 `MINIMAL_DIRS` 和 `STANDARD_EXTRA_DIRS` 常量。 这些应该最终来自 manifest：

```json
{
  "adoption": {
    "modes": {
      "minimal": {
        "directories": [...],
        "files": [...],
        "helpers": false
      },
      "standard": {
        "extends": "minimal",
        "directories": [...],
        "helpers": "compat-wrapper"
      },
      "self-host": {
        "extends": "standard",
        "runtimeSource": "repo"
      }
    }
  }
}
```

收益：

```text
[ ] CLI planner
[ ] docs
[ ] check-task-workflow
[ ] migration report
[ ] release tests
```

都可以读同一份 manifest，减少 drift。

## 4. Host Adapter / Tool Adapter 插件化

现在 Codex、Claude、CodeGraph、Waza、Brain 都已经有各自逻辑，但仍分散在命令层和 tools 层。0.6 之后可以抽象成统一接口：

```ts
interface HostAdapter {
  id: "codex" | "claude" | string;
  detect(ctx: HostContext): DetectionResult;
  planInstall(ctx: HostContext): Operation[];
  applyInstall(plan: Operation[]): ApplyResult;
  doctor(ctx: HostContext): Diagnostic[];
}

interface ToolAdapter {
  id: "codegraph" | "waza" | "brain" | string;
  check(ctx: ToolContext): Diagnostic[];
  planEnsure(ctx: ToolContext): Operation[];
  applyEnsure(plan: Operation[]): ApplyResult;
}
```

这样将来加 Cursor、Continue、Windsurf、Aider 或其他 host 时，不会污染 `index.ts`、`doctor.ts`、`global-runtime.ts`。

当前 process runner 已经统一了 `init/global-runtime/codegraph/helper-runner` 的子进程边界，说明适配层可以继续标准化。`init.ts` 和 `global-runtime.ts` 都已经使用 bounded runner。  CodeGraph 也已接入 bounded runner。

## 5. Adoption Planner 从 “file operation” 升级为 “domain operation”

目前 operation 主要是文件级：

```text
mkdir
writeFile
appendManagedBlock
mergeJson
move
remove
gitUntrack
runCheck
```

这是很好的底座。下一层可以引入 domain operation，再编译成 file operation：

```ts
type DomainOperation =
  | { kind: "installWorkflowContract" }
  | { kind: "ensureBootstrapDocs" }
  | { kind: "installHelperWrappers" }
  | { kind: "installReferenceConfigStubs" }
  | { kind: "retireLegacyRepoAdapters" }
  | { kind: "ensureGitignoreRuntimeBlock" };
```

然后：

```text
DomainOperation -> FileOperation[]
```

为什么有价值：review 层看 domain operation 更容易理解，apply 层仍保持 file operation 精确可 rollback。

JSON 可以同时输出：

```json
{
  "domainOperations": [
    {
      "id": "installWorkflowContract",
      "reason": "Make repo opt-in for hooks"
    }
  ],
  "operations": [
    {
      "id": "writeFile:.ai/harness/workflow-contract.json:workflow-contract"
    }
  ]
}
```

## 6. `setup check / doctor / security` 统一成 Readiness Pipeline

当前 `doctor`、`setup check`、`security scan` 有共同目标：检查状态、输出 remediation、保持 read-only。可以抽成：

```ts
interface ReadinessCheck {
  id: string;
  scope: "user" | "repo" | "host" | "tool" | "security";
  run(ctx: ReadinessContext): CheckResult;
  agentActions?(result: CheckResult): AgentAction[];
}
```

不同命令只是 renderer：

```text
repo-harness doctor             -> human renderer
repo-harness setup check --json -> agent-action renderer
repo-harness security scan      -> security-filtered renderer
repo-harness adopt --dry-run    -> adoption-readiness subset
```

这样不会出现 “doctor 检查一套、setup check 检查另一套、adopt 又提示另一套”。

## 9. Hook runtime 最终 typed-handler 化

现在 hook runtime 仍通过 shell scripts 执行，这有历史兼容价值。0.6 阶段不建议马上改。但中长期更优架构是：

```text
route registry -> typed handler id -> TS handler
```

例如：

```ts
{
  event: "PreToolUse",
  routeId: "edit",
  handlers: ["worktreeGuard", "preEditGuard"]
}
```

兼容策略：

```text
默认：package TS handler
repo pin：.ai/hooks/*.sh
self-host：repo-local handler
```

好处：

```text
[ ] 更快
[ ] 更可测
[ ] 更少 shell / platform 差异
[ ] 更强结构化错误
[ ] 更容易做 policy injection
```

但顺序上建议放在 rollback/apply 转正之后，不要和 adoption planner 迁移同时做。

## 10. 可观测性：给每次 apply / hook / helper 执行统一 run record

现在 adoption plan、process runner、helper runner 都逐步结构化了。下一步可以统一 runtime event schema：

```json
{
  "protocol": 1,
  "runId": "...",
  "command": "adopt",
  "phase": "apply",
  "repoRoot": "...",
  "startedAt": "...",
  "endedAt": "...",
  "operations": [...],
  "processes": [...],
  "result": "ok"
}
```

输出到：

```text
.ai/harness/runs/<run-id>.json
.ai/harness/runs/latest.json
```

这会让后续 agent handoff、doctor、failure diagnosis 更强。

## 最终判断

0.6 的架构已经不是“有没有 planner”的问题，而是进入了 **planner -> applicator -> transaction -> rollback -> default apply** 的产品化阶段。

当前最有价值的架构优化顺序是：

```text
1. 发布证据闭环
2. tarball install smoke
3. transaction manifest
4. rollback executor
5. domain operation layer
6. TS apply parity
7. default apply 切换
8. host/tool adapter 插件化
9. typed hook handlers
```

不要现在就重写 hook runtime 或拆 monorepo。0.6 刚打好 core/effects 边界，下一步应该优先把 **apply 的可靠性和可恢复性** 做扎实。
