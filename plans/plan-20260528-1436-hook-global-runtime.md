## Context

**Why this change.** 每个项目都要单独复制并维护 `.codex/hooks.json` (adapter) + `.ai/hooks/*.sh` (17 个脚本 + `lib/workflow-state.sh` 27.5K) + 每次跨项目还要在 Codex Settings 里重新 trust hooks，Claude Code 侧 `.claude/settings.json` 同理。这把 agentic-dev 的"按项目初始化"成本绑死在每个 repo 上，但 hook **实现** 本身和项目无关——它们只是在 `git rev-parse` 出 repo root 后调用 `.ai/hooks/<name>.sh`。

**痛点的真正形状是 runtime 重复，不是 state 重复。** 当前 `.codex/hooks.json` 的 command 完全是静态分派器，没有任何项目特定值；`.ai/hooks/run-hook.sh` 也只是个 dispatcher。这两层都可以 global 化。而 plan、todo、handoff、checks、architecture drift 是真正的 repo state，**不可以** global 化（并发 repo / worktree / 分支切换 / 审计归属会立刻坏掉）。

**预期结果。** 类 codegraph 模式：一个全局 `agentic-dev` CLI 同时安装到 Codex (`~/.codex/hooks.json`) **和** Claude Code (`~/.claude/settings.json`) 作为 hook 主入口；项目通过 opt-in marker (`.ai/harness/workflow-contract.json`) 告诉 CLI "我启用 agentic-dev workflow"；所有产物仍写当前 repo 内。新项目几乎不再有 hook 配置面；老项目通过渐进迁移路径过渡。

**CodeGraph 范式对照** (deep-read `_ref/codegraph@02935d77` 后**修正**, 详见 `tasks/notes/hook-global-runtime.notes.md`): `codegraph install --target <id> --location global` 写 `~/.codex/config.toml` 的 `[mcp_servers.codegraph]` 表 (`_ref/codegraph/src/installer/targets/codex.ts:40,46`)——**它注册的是 MCP server, 不是 hook, 完全不碰 `~/.codex/hooks.json`**。codegraph 的 git hooks (`src/sync/git-hooks.ts:72-80`) 只是 git post-event 触发 `codegraph sync` 增量索引, 不是通用 hook dispatcher。

**含义**: 我们借的是 **global installer + project-local state 的分层架构**, 以及 **multi-target registry pattern** (codegraph 实际支持 8 个 target: claude/codex/cursor/opencode/hermes/gemini/antigravity/kiro, `registry.ts:20-29`); 但**不能借 'codegraph 在生产中证明过 `~/.codex/hooks.json` 可加载'**——它根本不写这个文件。Codex global hook 文件机制的可加载性必须由我们 Phase 0 实测。已知事实 (`tasks/notes/codex-hook-adapter.notes.md:11`): `codex features list` reports hooks as stable enabled, 但 global vs project 维度仍未验证。Codex 仅支持 `--location global` (无 project-local, `codex.ts:57-59`)。

## Premises (Approved)

1. **`agentic-dev hook <event>` 是主入口。** Global hook 不直接调 repo shell；CLI 是唯一调度层。
2. **项目 opt-in 以 `.ai/harness/workflow-contract.json` (或 version marker) 为准。** CLI 在 repo root 里检测它；没有就 `exit 0` 静默 no-op。
3. **产物仍写项目内。** `.ai/harness/*`、`plans/*`、`tasks/*`、`tasks/notes/*` 全部 repo-local。
4. **`.codex/hooks.json` 和项目级 `.claude/settings.json` hook 段降级为兼容 fallback shim**，不再是长期主路径；installer 在新项目不再写它。
5. **第一刀先做 canary + installer/status，不直接大迁移。** 旧项目保持现状，迁移走可选路径。
6. **Multi-host installer。** CLI 暴露 `agentic-dev install --target codex|claude|both --location global`，同时配置两个 host 的 global hook 入口，让单一 CLI 在两个 agent runtime 下行为一致。

## Approaches Considered

### Approach A — Thin Global Adapter + Repo-Local Hook Implementations [SELECTED]

**Shape (multi-host extension).**

```
Codex event  ─┐
              ├─▶ agentic-dev hook <event>  ─▶  detect repo root
Claude event ─┘                                 ─▶  read .ai/harness/workflow-contract.json
                                                ─▶  exec <repo>/.ai/hooks/<mapped>.sh
                                                ─▶  hook writes .ai/harness/* / plans/* / tasks/*
```

- `~/.codex/hooks.json` 注册 6 行薄 adapter；`~/.claude/settings.json` 的 `hooks` 段同样注册相同事件。每行调 `agentic-dev hook <event>`。
- CLI 解析 repo root → 检测 opt-in marker → 找 `<repo>/.ai/hooks/<mapped-name>.sh` → exec。
- 项目仍保留 `.ai/hooks/*.sh` 实现（暂时——后续可演进为 sealed）。
- 新项目不再写 `.codex/hooks.json` 或 `.claude/settings.json` 的 hook 段；旧项目保留为 fallback。

**Why selected.**

- 痛点优先级匹配：最高频痛点是"开新项目时双 host 都要 trust hooks"，A + multi-host 一次性消掉两边。
- 与 codegraph 的 **分层架构 + multi-target registry** 对齐 (`_ref/codegraph/src/installer/targets/registry.ts:20-29`); **hook 加载路径本身仍需 Phase 0 实测**, codegraph 不证明此 (它写 MCP config, 不写 hooks.json)。
- 可降级：若 phase 0 canary 发现某 host 不加载 global hook，方案降级为该 host 走 fallback adapter，另一 host 仍受益。
- 改动小：CLI 入口 ~300 行 + adapter 各 6 行 + installer 加 `install --target`。可在 1-2 周 ship phase 0 + adapter。

**Trade-offs accepted.**

- 仍需复制 `.ai/hooks/*.sh`（实现层）— 这是 Approach B 的事，留作后续阶段（"Future Direction"）。
- 项目 hook 实现升级仍需逐项目同步——通过现有 `scripts/migrate-project-template.sh` 处理。
- Global adapter 和 project adapter 共存时，CLI 需自带优先级判定（global wins, project fallback only if CLI 不可用）。

### Approach B — Full Bundled CLI + Sealed Hook Implementations [DEFERRED]

把 17 个脚本 + 27.5K workflow-state.sh 全部打包进 CLI binary，项目里无 `.ai/hooks/`。彻底消除复制但工程量 L、风险 High、override 协议设计成本高。**作为 A ship 后的下一步（"Future Direction"），不在本 plan 范围。**

### Approach C — Canary-First Only [FOLDED INTO A]

C 的所有产物（双 host compatibility matrix、global hook 可加载性验证）作为 **Approach A 的 Phase 0** 自然产出。不需要作为独立方案。

## Recommended Approach (Final)

**Approach A + Multi-Host**，分两阶段执行:

### Phase 0 — Operational Smoke (0.5-1 天, 不再是 discovery)

**关键认知 (用户确认 + config.toml 实证后, 2026-05-28 第二轮)**:

两个 host 的 user-level hooks 加载性现已完全 documented + 实证:

- **Claude**: `~/.claude/settings.json` FULLY SUPPORTED + DOCUMENTED, 所有 events, merging precedence 清晰, 无 trust re-prompt (cite: code.claude.com/docs/en/settings.md & hooks.md)
- **Codex**: `~/.codex/hooks.json` 是 documented user-level path; `~/.codex/config.toml` `[features] hooks=true` + `[hooks.state]` 段已存在; trust hash key 用绝对路径格式, user-level path 一样接 hash
- **codegraph 不能借鉴的真正原因**: codegraph 早于 Codex hooks feature, 选 `config.toml` MCP 段是当时唯一可用配置面; 不是因为 hooks.json 不可行

**Phase 0 真实目标 (大幅缩水)**:

目标: 实测两 host 的 trust prompt UX + hash 注册行为, 把发现记录到 `docs/architecture/global-hook-runtime.md` Host Operational Matrix。Phase 0 可与 Phase 1 实施重叠 (写完 canary 立即推进 CLI 主线)。

- 写一个 noop canary hook，echo `[agentic-dev canary] host=$HOST event=$EVENT repo=$(git rev-parse --show-toplevel)`

## What I noticed about how you think

- 你**先识别 codegraph 的范式再套用过来**，而不是从 hook 系统设计原则推导。这种 "看到的方案 → 抽象成模式 → 应用到新问题" 的复用路径在工程上效率最高，但也意味着如果 codegraph 的某个隐含假设不成立（比如它的 install 时机、host 加载行为），方案也会跟着错。Phase 0 canary 实际上就是在抵御这个风险。

- 你说"产物是项目级"是天然的，没把它当 design space。这是对的；但值得在 future direction 里留意：cross-repo aggregation 一旦做，那个"项目级"边界会回来。
